# Compte Rendu d'incident - CRIM-08-05-2026

## Contexte

Le 8 mai 2026, j'ai voulu lancer une mise a jour de talos vers la version 1.12.7 depuis la version 1.12.6. Cependant, j'ai merdé et pas prévu qu'en changeant la version de talos dans la version existante du script, celui-ci générerait une nouvelle url d'installation et engendrerait une réinstallation compète du noeud.

## Impact

- Le noeud a été réinstallé et a perdu toutes les données qui n'étaient pas sauvegardées.
- Le cluster a été indisponible pendant la durée de l'installation.
- Réinstallation de ZERO impliquant la création de ce fichier ainsi que la création par conséquent d'un processus étape par étape pour réinstaller le cluster (Ouais dans mon malheur je vais pouvoir creer plus de documentation pour les prochaines fois).

## Résolution

- Ajout d'une nouvelle variable dans le script pour différencier la version de talos utilisée pour l'installation et la version de talos utilisée pour la configuration du cluster.
- Mise en place d'un step by step pour mettre a jours le cluster sans réinstaller les noeuds.
- Mise en place d'un step by step de disaster recovery pour réinstaller le cluster en cas de besoin.
- Désactiver l'auto approve des commandes pulumi pour éviter de faire des erreurs de ce genre à l'avenir.
- Planifier la mise en place de backups réguliers pour éviter de perdre des données à l'avenir.
  - Certain service ne nécessitent pas de backup au vue de l'approche GitOps mise en place ainsi que la capacité a recréer une grande partie des données assez facilement (ex: Forgejo, Vault) mais d'autre service comme Vaultwarden nécessitent des backups réguliers pour éviter de perdre des données critiques.

## Disaster recovery sans Backup

0. Vider l'état pulumi si il existe ( rm -rf .tmp/pulumi)
1. `task pulumi:up` pour réinstaller le noeud avec la nouvelle version de talos.
2. Switch 0.pulumi/Pulumi.dev.yaml 0.pulumi-conf:deployed a false pour ne pas déployer l'ingress de l'argo
3. `task pulumi-config:up` pour réappliquer la configuration du cluster. (attendre que le booting sois finis via `task talosctl -- dashboard` pour voir l'état du noeud)
4. Make sure that each app is started with the token (disable registry, etc and wait for vault to be up and running)
5. Manually create a VPN token and create it with `task vault:vpn TOKEN=xxx` (or use the vault cli if you have access to the vault instance)
6. Manually create a forgejo token and create it with `task vault:git API_TOKEN=xxx` (or use the vault cli if you have access to the vault instance)

## Troubleshooting

### Forgejo : vault {"errors":["could not load configuration"]}

- Delete forgejo auth backend and rerun the terra-vault task

### Rybbit : Can't register user

- Enable register in chart and disable it after registering

## Bilan

- Interuption de service du cluster de 00H30 a 3H20 (heure de retour de monofolio)
